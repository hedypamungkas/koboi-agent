## 0. Internal Snapshot

### What it does

koboi-agent is an open-source, self-hostable, YAML-driven, configurable AI-agent framework (async Python 3.10+, alpha v0.4.0, ~31.6k LOC / 187 .py source files / ~161 test files / ~2,900 test functions). A user authors a YAML config and gets a chat/act agent with tools, RAG, guardrails, multi-provider LLM, MCP, skills, sandboxing, multi-agent orchestration, and evals — runnable four ways: (1) Python SDK via the single KoboiAgent facade (from_config/from_dict/from_config_string + run/run_stream/resume/close); (2) CLI/TUI (`koboi chat|run|sessions|serve|keys|eval|eval-test|validate|diagnostics|init-zsh`); (3) a Textual terminal UI (9 screens, 13 widgets); (4) a FastAPI HTTP/SSE server (`koboi serve` or create_app()) exposing interactive streaming chat with human-in-the-loop SSE approvals PLUS autonomous background jobs (202 + poll/replay, deny-by-default on destructive tools, restricted-sandbox-mandatory). 18 top-level YAML sections with ${VAR:default} interpolation and `extends` inheritance — the YAML file IS the agent's primary UX. Cite: pyproject.toml:7-14; README.md:1-83; koboi/facade.py:45-141; koboi/server/CLAUDE.md:33-48; docs/architecture.md:284-306.

### Differentiators (the things worth benchmarking)

- YAML-first configuration UX (18 sections, ${VAR:default} env interpolation, `extends` inheritance) — a 16-line simple_chat.yaml yields a working agent, and one config family spans dev/prod. The config file is the product surface. Cite: configs/simple_chat.yaml; docs/architecture.md:284-306.
- Durability + resume: SQLite-backed memory PLUS a per-iteration StepJournal ('steps' table, eager 'running' markers committed before each LLM call) enabling `koboi run --resume <session>` and jobs `resume_on_startup` to rehydrate interrupted sessions across crashes/redeploys. Most agent frameworks have no equivalent crash-recovery substrate. Cite: koboi/journal.py:62-135; koboi/loop.py:357-436, 495-514.
- Two execution modes in ONE serving layer sharing one AgentPool: interactive SSE chat (HITL via AsyncCallbackApprovalHandler → PendingApprovalEvent → POST /v1/sessions/{id}/approve, 120s timeout→deny) AND autonomous background jobs (202, deny-by-default AutonomousApprovalHandler, restricted sandbox mandatory, resume_on_startup). Cite: koboi/server/CLAUDE.md:50-57; koboi/server/app.py:142-345.
- Deep, layered safety stack: input/output guardrails + PolicyEngine + ApprovalHandler (CLI/callback/auto) + TrustDatabase (graduated permissions) + restricted sandbox (per-session workdir, PATH allowlist, rlimits via preexec_fn, secret-env scrubbing, network deny) + HITL SSE approvals + deny-by-default autonomous jobs + YOLO opt-in only (jobs always reject yolo). Cite: koboi/sandbox/restricted.py:89-200; koboi/harness/env.py:58-134; koboi/server/CLAUDE.md:74-90.
- Consistent registry/observer extension model with 10+ uniform seams and 4 plugin entry-point groups (koboi.providers/tools/guardrails/scorers) discovered at import — every extension point is ABC + registry + @decorator, so external packages add capabilities without forking. Cite: docs/architecture.md:336-377; pyproject.toml:54-68; koboi/plugins.py.
- Multi-provider LLM (OpenAI default, Anthropic, Cloudflare) via ProviderRegistry + entry-point plugins, with a RetryClient doing exponential backoff honoring Retry-After and a streaming retry guard that will not resume a partial stream. Cite: koboi/client.py:121-175; koboi/llm/registry.py.
- Two-paths-same-composition serving: zero-code `koboi serve <config>` vs programmatic create_app(config, extra_tools=, extra_hooks=, approval_handler=) — gives adopters a graduated complexity curve and a single shared apply_tool_selection() so selection semantics cannot drift. Cite: README.md:79-82; koboi/tools/registry.py:303-367; koboi/orchestration/factory.py:289-293.
- Self-host deployment artifacts are real and least-privilege: Dockerfile (USER 1000, cap_drop ALL, no-new-privileges, healthcheck on /healthz), docker-compose.yml with Cloudflare Tunnel + Langfuse wiring, nginx alternative, plus a 3-tier extension story (config-mount / extension-dir / derived-image). Cite: Dockerfile:1-45; docker-compose.yml:1-34.
- Lean 4-dependency core (httpx, pyyaml, python-dotenv, pydantic v2) with all heavy capabilities (FastAPI, Textual, Langfuse, eval suites) gated behind optional extras — minimal install surface for library consumers. Cite: pyproject.toml:21-49.
- Production hygiene unusual for an alpha: graceful drain (cancel streams → flush Langfuse off-loop+concurrent → cancel jobs → close pool/stores under wait_for timeout), Idempotency-Key 409-reject/dedup, per-session ownership, Bearer auth with rotate/revoke (SHA-256 hashed, hmac.compare_digest), /readyz pool+DB ping, X-Request-Id mint/honor/echo. Cite: koboi/server/app.py:229-266; koboi/server/auth.py:72-130; koboi/server/health.py:54-67.

### Strengths

- Explicit, clean layering: pure-data layer (types.py, events.py, exceptions.py, tokens.py) → primitives (config.py, memory.py, hooks/chain.py, tools/registry.py, modes.py) → subsystems (client.py, llm/, rag/, context/, guardrails/, harness/, sandbox/, skills/, mcp/, journal.py, trust.py) → loop.py (AgentCore) + loop_pipeline.py → facade.py → top-level surfaces. Cite: docs/architecture.md:9-60.
- Facade + AgentAssembler pattern runs 14 build steps in dependency order and exposes each result (logger, client, memory, tools, hook_chain) as an inspectable attribute, making the dependency graph legible. Cite: koboi/facade.py:124-141, 722-965.
- Observer-pattern hook system cleanly decouples cross-cutting concerns (logging, policy, telemetry, guardrails, carryover, doom-loop detection, skill persistence) from the core loop across 15 priority-sorted events; AgentInfo prevents bidirectional coupling. Cite: koboi/hooks/chain.py:48-60, 227-298.
- Typed, layered error design: distinct LLMError and AgentError hierarchies mapped at the transport boundary (HTTP status → Auth/RateLimit/InvalidRequest/Server/Connection/ResponseParse), so callers catch by intent rather than parsing strings. Cite: koboi/llm/base.py:15-43; koboi/exceptions.py:6-39; koboi/llm/http_transport.py:113-137.
- Defense-in-depth at every server boundary: session_id validated at route AND in workdir_for() (regex ^[A-Za-z0-9_-]{1,128}$); auth fails closed when no keys + auth_required; jobs reject yolo and require sandbox.backend='restricted'; output-guardrail buffering prevents blocked-token leakage (TextDeltas buffered until _process_output passes). Cite: koboi/server/pool.py:75-135; koboi/loop.py:525-577.
- Defense-in-depth auth + sandbox: SHA-256 hashed keys + hmac.compare_digest constant-time compare; restricted backend combines path containment, rlimits-in-child, PATH allowlist, proxy/NETRC strip, secret-shape env blocklist. Cite: koboi/server/auth.py:72-130; koboi/sandbox/restricted.py:89-200; koboi/harness/env.py:58-134.
- Mature release pipeline: tag-triggered, OIDC trusted PyPI publishing (no long-lived credentials), multi-version test gate (3.10/3.11/3.12) before publish; CI enforces ruff+format+mypy, --cov-fail-under=80, bandit hard gate. Cite: .github/workflows/release.yml:54-68; .github/workflows/ci.yml:34-59.
- Honest, code-grounded documentation: CLAUDE.md and per-subpackage CLAUDE.md files explicitly call out load-bearing gotchas (AgentCore not concurrent-safe, sandbox workdir was once global, agent.close() does not flush Langfuse, M5 seams intentionally not pre-protocol-ized), reducing operational surprises. Cite: CLAUDE.md (root); koboi/server/CLAUDE.md; koboi/sandbox/CLAUDE.md.
- Streaming correctness details handled: stream_options.include_usage requested so token accounting is not null on streamed responses; two execution paths (interactive + jobs) share one AgentPool and one apply_tool_selection(). Cite: koboi/llm/openai_adapter.py:82-84; koboi/tools/registry.py:303-367.
- Well-isolated testable wire layer: pure-Pydantic schema.py (no FastAPI import) and pure sse.py encoder let the SSE wire format be unit-tested without the [api] extra; asyncio_mode=auto with MockClient + shared fixtures across ~2,900 tests. Cite: koboi/server/schema.py:1-4; tests/conftest.py:16-118.
- Bundled runnable artifacts lower time-to-value: 9 builtin tools, 4 sample skills, 1 MCP server (todo_server.py), Acme-Corp sample RAG corpus, BFCL eval data, 32 numbered examples + 20 configs. Cite: .claude/rules/builtin-tools.md; skills/; mcp_servers/todo_server.py; data/sample/.
- Langfuse tracing self-disables when keys are empty or the package is absent (fail-open), so the same config is safe in dev (no tracing) and prod (tracing on). Cite: koboi/hooks/langfuse_hook.py:22-28; configs/e2e_full.yaml:131-133.

### Limitations competitors likely exploit

- NO first-party web UI and NO first-party HTTP client SDK — the single biggest gap between ambition and out-of-box experience. The 'API platform' pitch is server-side only; every consumer must hand-roll curl/httpx + SSE parsing + approval polling (/v1/sessions/{id}/approve) + job-status polling. Competitors with bundled UIs/SDKs exploit this heavily. Cite: docs/one-pager-api-platform.md; docs/rest-sse-requirements.md:239-273; README.md:56-83.
- Single-node, SQLite-only persistence: ONE koboi_memory.db (WAL, busy_timeout=5000) holds messages, sessions, steps (journal), plus sidecar session_owners and jobs tables — one writer is the persistence SPOF. AgentPool (cap=100, LRU eviction, PoolFull→429) and IdempotencyRegistry are in-memory only (lost on restart). M5 Protocols (SessionStore/LockProvider/EventBuffer) are forward-only stubs with no Redis/SaaS backend — no horizontal-scaling story today. Cite: koboi/memory_sqlite.py:54-101; koboi/server/app.py:129-193; koboi/server/protocols.py; koboi/server/pool.py:86-118.
- AgentCore is NOT concurrent-safe by construction, forcing per-session serialization (one in-flight turn per session via asyncio.Lock held for the full stream lifetime) and a per-request snapshot/restore discipline (mode/max_iterations/_tool_pipeline/approval_handler) that is easy to forget — a per-session throughput ceiling and a recurring integration tax on every embedding path. Cite: koboi/server/pool.py:1-16, 207-236; koboi/server/CLAUDE.md:74-86.
- No OpenAI-compatible adapter in v1 — the API is koboi-native SSE+jobs; OpenAI-compat is explicitly deferred. This blocks drop-in use of the large ecosystem of OpenAI SDK clients and is a recurring ask for SaaS adoption. Cite: docs/rest-sse-requirements.md:23, 89.
- CLI command surface is split across install extras and silently fails: `koboi chat/run/sessions/eval/validate/diagnostics/init-zsh` are defined in the TUI Click group installed only with the [tui] extra; `koboi serve`/`keys` need [api]. A bare `pip install koboi-agent` then `koboi chat ...` prints ImportError and exits, but the README Quickstart never explains this. Cite: koboi/cli.py:91-114; koboi/tui/app.py:125-448.
- Restricted sandbox network denial is explicitly SOFT — blocks obvious egress binaries (curl/wget/nc) but not interpreters (python3 -c 'import urllib'). True network/container isolation requires an UNBUILT Docker/gVisor backend (P0c), so autonomous jobs on untrusted prompts remain a risk. Cite: koboi/sandbox/restricted.py:10-13, 130-140; koboi/sandbox/CLAUDE.md.
- Authoritative product docs are mixed English/Indonesian (docs/rest-sse-requirements.md headings like 'Konteks & Tujuan', 'Konkurensi', 'Dwi-lapis') and docs/one-pager-api-platform.md describes a superseded NATS+multi-container design never built — an English-speaking open-source adopter landing on these is misled. Cite: docs/rest-sse-requirements.md (headings); docs/one-pager-api-platform.md:1-10.
- Install-extras fragmentation harms onboarding: dev, tui, api, tracing, plus five separate eval-* extras (eval-ragas/eval-gaia/eval-swe-bench/eval-deepeval/eval-all) — no obvious 'just try the CLI' one-liner matching the documented Quickstart. Cite: pyproject.toml:33-49.
- Alpha maturity declared (Development Status :: 3 - Alpha, v0.4.0) — the API surface, SSE event protocol, and error-code enum should be treated as unstable, which is awkward given the product targets self-host production deploys. Cite: pyproject.toml:7,14.
- Production-default CORS is allow_origins: ['*'] in the reference server_deploy.yaml (only an inline comment to lock it down), and the Dockerfile's default CMD config is configs/e2e_full.yaml which the file itself labels an e2e test fixture — both copy-paste footguns for self-hosters. Cite: configs/server_deploy.yaml:48-51; Dockerfile:22,45; configs/e2e_full.yaml:1-3.
- Module-global singletons break multi-agent isolation in one process: harness/env.py holds _env_defaults (last-built config wins — flagged in code comment), and MemoryStore paths can collide. This affects orchestration mode (multiple sub-agents in one process). Cite: koboi/harness/env.py:82-88; koboi/tools/registry.py:348-354.
- Config is a plain dict at runtime; dotted-path config.get('server','auth_required',default=True) is the runtime truth and Pydantic ServerConfig/JobsConfig are explicitly 'cosmetic validation only' — schema can drift from code silently so misconfigured YAML can pass parse yet fail later. Cite: koboi/server/CLAUDE.md:62-66; koboi/config.py:122-130.
- Simplified job resume: running jobs are marked failed-on-restart (running→failed InterruptedByRestart, pending→requeued) rather than journal-rehydrated — full resume deferred to M5, so long-running autonomous jobs lose progress on every redeploy. Cite: koboi/server/jobs.py:6-8.
- Two parallel agent-build paths (AgentAssembler in facade.py vs AgentFactory/_build_tools_from_config in orchestration/factory.py) duplicate builtin-register + memory-inject + keep_only logic; only apply_tool_selection is shared, so registration-time drift remains possible. Cite: koboi/facade.py:459-494 vs koboi/orchestration/factory.py:254-294.

## 1. Executive Summary

koboi-agent (v0.4.0 alpha, ~31.6k LOC / 187 source files / ~2,900 tests) is an open-source, self-hostable, YAML-driven, configurable AI-agent framework (async Python 3.10+) with a genuine, unusual differentiator: a BUNDLED durable serving layer that puts interactive streaming chat with human-in-the-loop SSE approvals AND autonomous deny-by-default background jobs (with crash/redeploy resume) behind one config file (docs/rest-sse-requirements.md:239-273; koboi/server/CLAUDE.md:33-57; koboi/journal.py:62-135). Its feature set maps almost 1:1 onto the developer community's own "production-ready agent framework" checklist (HN story 46998170: persistent memory, tool use with error recovery, multi-model swap, plugin/skill extensibility, runs-as-daemon, security boundaries) — the single strongest product-market-fit signal in the project.

Three structural realities shape strategy. (1) MINDSHARE ASYMMETRY: every direct rival has 18k-141k GitHub stars (verified 2026-07-04/05 via GitHub REST API: LangChain 141k, AutoGen 59.5k, CrewAI 54.9k, LlamaIndex 50.6k, LangGraph 36.5k, smolagents 28.2k, OpenAI Agents SDK 27.6k, Mastra 25.8k, Vercel AI SDK 25.3k, Pydantic AI 18.2k) vs koboi's unlisted alpha; adoption, not capability, is the first-order threat. (2) DIFFERENTIATION EROSION: Pydantic AI now ships a YAML/JSON agent-spec + durable execution (eroding the YAML-first claim); OpenAI Agents SDK ships container-based Sandbox Agents (exposing koboi's SOFT network sandbox); Mastra/Vercel/CrewAI all ship first-party UIs koboi entirely lacks. (3) TRUST TROUGH WINDOW: developer usage is up (84% use/plan, 51% of pros daily) but sentiment fell 70%+ → 60% and 46% distrust AI accuracy (Stack Overflow 2025) — a disillusioned market rewards exactly koboi's reliability-first, observable, HITL-by-default posture.

The strategic thesis: koboi cannot win on agent-loop ergonomics or mindshare. It should win on the SPECIFIC BUNDLE of Python + YAML-first config + self-hostable + dual-mode durable serving + least-privilege Docker — operationalized as "the agent framework built for the 76% who won't yet trust agents in production" (Stack Overflow 2025: 76% won't use AI for deployment/monitoring). The two highest-leverage moves are (a) ship an OpenAI-compatible adapter on top of the durable server (defensible 2026 wedge, unblocks the largest client ecosystem) and (b) ship a first-party client SDK + minimal web UI (closes the biggest out-of-box gap). Without both, koboi's strongest features remain un-demoable without writing client code.

## 2. Competitive Landscape

koboi-agent competes in a crowded "build + serve AI agents" category where every direct rival has 10x-200x its mindshare (verified 2026-07-04/05 GitHub stars: LangChain 140.9k, AutoGen 59.5k, CrewAI 54.9k, LlamaIndex 50.6k, LangGraph 36.5k, smolagents 28.2k, OpenAI Agents SDK 27.6k, Mastra 25.8k, Vercel AI SDK 25.4k, Pydantic AI 18.2k — vs koboi's unlisted alpha). The competitive set splits into (a) Python orchestration-first frameworks (LangGraph, OpenAI Agents SDK, Pydantic AI, CrewAI, AutoGen, LlamaIndex, smolagents), (b) TypeScript serving-first SDKs (Vercel AI SDK, Mastra). koboi's "YAML-first config + bundled durable serving layer" positioning is its only unoccupied lane, but it is no longer unique: Pydantic AI now ships a YAML/JSON agent-spec + durable execution, OpenAI Agents SDK ships Sandbox Agents, and Mastra ships a full Studio UI + suspend/resume + cloud — each closing one of koboi's claimed differentiators. koboi's two biggest gaps against the field are universal: (1) no first-party web UI / client SDK (Mastra Studio, Vercel AI SDK UI, OpenAI Agents UI, LangSmith Studio, CrewAI Studio all exist) and (2) no OpenAI-compatible API (Vercel, OpenAI SDK, Mastra, Pydantic AI all interoperate with the dominant client ecosystem). The window to defend "YAML-first self-hostable agent server" is narrowing in 2026 as incumbents add config layers and serving/cloud tiers.

## 3. Market Opportunities

koboi-agent enters a market that is simultaneously booming and sobering. The AI agents market is projected at USD 52.62B by 2030 (46.3% CAGR, MarketsandMarkets Apr-2025), GitHub saw a 59% surge in generative-AI project contributions and a 98% rise in AI projects in 2024 (Octoverse), and 84% of developers now use/plan AI tools with 51% of professionals using them daily (Stack Overflow 2025). Yet sentiment has cooled from 70%+ (2023-24) to 60% (2025), 46% actively distrust AI accuracy vs 33% who trust it, and 76% refuse to let AI touch deployment/monitoring. The center of gravity has shifted from "demo magic" to "production reliability," which is exactly where an opinionated, YAML-first, self-hostable framework with bundled durable serving can win. Koboi's feature set maps almost 1:1 onto the developer community's own "production-ready agent framework" checklist (HN 46998170): persistent memory across sessions, real tool use with error recovery, multi-model swap, plugin/skill extensibility, runs-as-a-daemon (FastAPI server + jobs), and security boundaries (sandbox/guardrails/audit) — the commenter's complaint that "most frameworks nail 1-2 of these but fall apart on the rest" is koboi's precise wedge. The buyer/user split is developer-led bottom-up: the USER (backend/AI/platform engineers, startup CTOs) is also the initial BUYER via OSS adoption, but enterprise self-host adoption pulls the buyer up to engineering managers / platform leads / CTOs with security and compliance as gatekeepers. Three demand drivers favor koboi specifically: (1) data-sovereignty / EU-AI-Act pressure pushing regulated industries (finance, healthcare, government, legal) toward self-hostable agent runtimes; (2) the vertical-AI-agent thesis (a16z/TechCrunch: "vertical AI agents could be 10x larger than SaaS"), where koboi's YAML-config-as-product-surface lets a vertical builder ship a domain agent without forking; (3) MCP standardization, which koboi already speaks. The biggest market-side gaps to close are the absence of a first-party UI/client SDK, no OpenAI-compatible adapter (blocks the large drop-in ecosystem), single-node SQLite (no horizontal-scale story), and alpha maturity declared against production ambitions — all recurring buying-criteria blockers for the enterprise segment koboi's feature depth otherwise targets.

## 5. Product Gaps

1. NO first-party web UI and NO first-party HTTP client SDK — the single biggest gap between ambition and out-of-box experience. Every consumer must hand-roll curl/httpx + SSE parsing + approval polling (/v1/sessions/{id}/approve) + job-status polling. Competitors with bundled UIs/SDKs (Mastra Studio, Vercel AI SDK UI, CrewAI Studio, AutoGen Studio, LangSmith Studio, OpenAI Agents tracing UI) exploit this heavily. (docs/one-pager-api-platform.md; docs/rest-sse-requirements.md:239-273)
2. NO OpenAI-compatible adapter in v1 — the API is koboi-native SSE+jobs; OpenAI-compat is explicitly deferred (docs/rest-sse-requirements.md:23,89). Blocks drop-in use of the large ecosystem of OpenAI SDK clients (Python/JS/Go/Rust) and is a recurring, community-documented ask (HN 'Nitro' 149 pts; 'Open Responses' 2025-04-02).
3. Single-node, SQLite-only persistence — ONE koboi_memory.db (WAL, busy_timeout=5000) holds messages, sessions, steps, sidecar session_owners and jobs tables; one writer is the persistence SPOF. AgentPool (cap=100, LRU, PoolFull→429) and IdempotencyRegistry are in-memory only (lost on restart). M5 Protocols (SessionStore/LockProvider/EventBuffer) are forward-only stubs with NO Redis/SaaS backend — no horizontal-scaling story today. (koboi/memory_sqlite.py:54-101; koboi/server/app.py:129-193; koboi/server/protocols.py)
4. Restricted sandbox network denial is explicitly SOFT — blocks curl/wget/nc but not interpreters (python3 -c 'import urllib'), the exact bypass class documented in 'Observed Agent Sandbox Bypasses' (HN 66 pts, Dec-2025). True container/gVisor isolation (P0c) is UNBUILT, so autonomous jobs on untrusted prompts remain a risk while competitors (OpenAI Sandbox Agents, E2B, Microsandbox, Tilde.run) ship real container isolation. (koboi/sandbox/restricted.py:10-13,130-140)
5. AgentCore is NOT concurrent-safe by construction, forcing per-session serialization (one in-flight turn per session via asyncio.Lock held for the full stream lifetime) plus a per-request snapshot/restore discipline (mode/max_iterations/_tool_pipeline/approval_handler) that is easy to forget — a per-session throughput ceiling and a recurring integration tax on every embedding path. (koboi/server/pool.py:1-16,207-236; koboi/server/CLAUDE.md:74-86)
6. Module-global singletons break multi-agent isolation in one process: harness/env.py holds _env_defaults (last-built config wins — flagged in code comment), and MemoryStore paths can collide. Directly affects orchestration mode (multiple sub-agents in one process) and undercuts koboi's multi-agent claim vs AutoGen/CrewAI/LangGraph. (koboi/harness/env.py:82-88; koboi/tools/registry.py:348-354)
7. Two parallel agent-build paths (AgentAssembler in facade.py vs AgentFactory/_build_tools_from_config in orchestration/factory.py) duplicate builtin-register + memory-inject + keep_only logic; only apply_tool_selection is shared, so registration-time drift remains possible. (koboi/facade.py:459-494 vs koboi/orchestration/factory.py:254-294)
8. CLI command surface is bifurcated and silently fails: `koboi chat/run/sessions/eval/validate/diagnostics/init-zsh` live in the TUI Click group installed ONLY with the [tui] extra; `koboi serve`/`keys` need [api]. A bare `pip install koboi-agent` then `koboi chat ...` prints ImportError and exits, but the README Quickstart never explains this. (koboi/cli.py:91-114; koboi/tui/app.py:125-448)
9. Install-extras fragmentation harms onboarding: dev, tui, api, tracing, plus five separate eval-* extras (eval-ragas/eval-gaia/eval-swe-bench/eval-deepeval/eval-all) — no 'just try the CLI' one-liner matching the documented Quickstart. (pyproject.toml:33-49)
10. Alpha maturity declared (Development Status :: 3 - Alpha, v0.4.0) — the API surface, SSE event protocol, and error-code enum should be treated as unstable, which is awkward given the product targets self-host production deploys. (pyproject.toml:7,14)
11. Production-default CORS is allow_origins: ['*'] in configs/server_deploy.yaml (only an inline comment to lock it down), and the Dockerfile's default CMD config is configs/e2e_full.yaml which the file itself labels an e2e test fixture — both copy-paste footguns for self-hosters. (configs/server_deploy.yaml:48-51; Dockerfile:22,45; configs/e2e_full.yaml:1-3)
12. Config is a plain dict at runtime; dotted-path config.get('server','auth_required',default=True) is the runtime truth while Pydantic ServerConfig/JobsConfig are explicitly 'cosmetic validation only' — schema can drift from code silently so misconfigured YAML can pass parse yet fail later. (koboi/server/CLAUDE.md:62-66; koboi/config.py:122-130)
13. Simplified job resume: running jobs are marked failed-on-restart (running→failed InterruptedByRestart, pending→requeued) rather than journal-rehydrated — full resume deferred to M5, so long-running autonomous jobs lose progress on every redeploy. (koboi/server/jobs.py:6-8)
14. Authoritative product docs are mixed English/Indonesian (docs/rest-sse-requirements.md headings 'Konteks & Tujuan', 'Konkurensi', 'Dwi-lapis') and docs/one-pager-api-platform.md describes a superseded NATS+multi-container design never built — an English-speaking open-source adopter landing on these is misled.

16. Runtime artifacts committed at repo root (koboi_memory.db 2.8MB SQLite+WAL, .agent_memory.json 159KB, .coverage 100KB, coverage_baseline.json 566KB) bloat the repo, risk secrets, and collide with clean checkouts; .gitignore hygiene + a default data/ path needed.

**Evidence:** docs/rest-sse-requirements.md:23,89 (koboi-native protocol, OpenAI-compat deferred); HN 'Nitro - OpenAI-Compatible API' 2024-01-06 pts=149; HN 'Open Responses' 2025-04-02; competitive research: Vercel AI SDK + Mastra + OpenAI SDK + Pydantic AI all interoperate OpenAI-compatible protocols.

> **Rationale:** Real reliability smell but overstated: validation already runs by default and unknown keys are already warned, so the "passes parse, fails at runtime" window is really just an independent schema-coverage gap; true fail-fast needs extra="forbid" which fights the framework's open extensibility, and the hidden cost is redesigning live config.raw mutation (TUI model swap, server pool per-session fork) that the immutable Pydantic snapshot can't serve.

> **Biggest risk:** Pydantic-as-truth is incompatible with two existing runtime paths — config.raw mutation for live model/provider swaps and per-session config forking in the server pool — and delivering the headline "fail at parse" requires extra="forbid" which breaks custom-key/forward-compat extensibility, so the refactor either can't fully deliver fail-fast or forces a live-mutation redesign that balloons scope to 5-7 PW.

### 10. Extract M5 Protocols into real Redis backends (SessionStore + distributed LockProvider + EventBuffer) for horizontal scaling

**Category:** Infra · **Complexity:** high · **Effort estimate (proposer):** 6-10 person-weeks. Implements RedisSessionStore + RedisLockProvider + RedisEventBuffer + cross-node IdempotencyRegistry, plus multi-node integration tests (node-kill, restart, affinity-free routing) and ops docs.

**Problem.** Single-node SQLite (one koboi_memory.db, one writer) plus in-memory AgentPool (cap=100 LRU) and in-memory IdempotencyRegistry (lost on restart) mean koboi has NO horizontal-scaling story — a recurring enterprise disqualifier. AgentCore is not concurrent-safe by construction, forcing per-session serialization with no cross-node coordination. Observable symptom: a team needing HA or >100 concurrent sessions cannot deploy koboi across multiple nodes; a node restart drops idempotency dedup and pool state.

**Why it matters.** This is the prerequisite for any FedRAMP/SAM-adjacent enterprise bid where CrewAI (claims 63% Fortune 500 + SAM/FedRAMP High) and LangGraph (managed Deployment with horizontal scaling) currently own the segment unopposed. The seams already exist as forward-only M5 Protocol stubs — implementing real Redis backends unlocks multi-node deploys and survives restarts without rebuilding agents from SQLite, converting a known scaling ceiling into an enterprise-credible runtime.

**Supporting evidence:**
- koboi/server/protocols.py:1-53 (verified) — forward-only SessionStore/LockProvider/EventBuffer Protocol stubs with NO Redis implementation; docstring explicitly defers to 'a future Redis-backed or multi-tenant implementation'
- koboi/server/pool.py (cited via report + koboi/server/CLAUDE.md) — in-memory AgentPool, cap=100, LRU eviction, PoolFull->429; per-session asyncio.Lock held for stream lifetime (not concurrent-safe)
- koboi/memory_sqlite.py (verified, 311 lines) — single-node SQLite WAL, busy_timeout=5000, one writer
- Competitor research (report, tag as research-sourced): CrewAI 63% Fortune 500 + SAM/FedRAMP High; LangGraph managed Deployment with horizontal scaling

**Expected customer impact.** Multi-node deploys with session affinity-free routing; restart survives without rebuilding agents from SQLite; distributed idempotency dedup across nodes.

**Expected business impact.** Enterprise credibility; unblocks the regulated-industry/data-sovereignty wedge; the load-bearing prerequisite for any serious enterprise procurement bid.

**Risks:** Redis becomes a new ops dependency and SPOF — offer Redis Sentinel/Cluster guidance and keep SQLite as the single-node default; Distributed-lock correctness (deadlocks, fencing tokens, crash recovery) is subtle — use a battle-tested recipe (Redlock with fencing or a lease table); Event-buffer ordering under concurrency; AgentCore per-session serialization still caps per-session throughput even after this lands
**Dependencies:** Redis (or Redis-compatible SaaS) in the deploy environment; AgentCore concurrency model is unchanged — this enables multi-node, not intra-session concurrency
**Success metrics:** 2-node deploy passes a session-affinity-free routing test (a request hits either node for the same session); Distributed lock survives a simulated node kill (no double-execution, no deadlock); IdempotencyRegistry survives a restart cross-node (same Idempotency-Key -> same response)

> **Adversarial score** — impact **5/10**, effort **8/10**, ROI **0.6**, confidence **0.55**, strategic alignment **6/10** (short-term 3, long-term 7).
>
> **Rationale:** Real gap with clean M5 seams, but the 6-10 PW estimate undercounts hidden work (fencing tokens through loop.py, route-handler refactor off record.events, multi-node integration-test infra) and the load-bearing prerequisite for true affinity-free routing — a distributed ConversationMemory/shared-storage story for the local-file koboi_memory.db — is unscoped and not listed as a dependency, so the initiative as written cannot deliver its own success metric.
>
> **Biggest risk:** KoboiAgent's conversation memory and per-session sandbox workdir are local to a node (memory_sqlite.py opens a local sqlite3 file; workdir_for is ./workspace/<id> on local disk), so even with Redis backends a request for session X landing on node B rebuilds an agent with an empty workdir and a foreign/empty koboi_memory.db — affinity-free routing silently fails unless distributed memory is also built, which is out of scope.

### 11. Add opinionated A2A (Agent-to-Agent) protocol support and fix multi-agent-in-one-process isolation

**Category:** Integration · **Complexity:** high · **Effort estimate (proposer):** 4-8 person-weeks. Isolation fix ~2-3w (singleton removal + per-agent config plumbing + isolation tests); A2A agent-card + task-send/streaming ~2-5w as an opt-in orchestration transport.

**Why it matters.** Early opinionated A2A support is a low-cost mindshare bet that could differentiate koboi before the protocol market settles, and shipping it cleanly forces resolution of the multi-agent isolation gap as a side effect — turning a known limitation into contested-protocol leadership. The isolation fix is the load-bearing deliverable; A2A is the opt-in upside on top of a correct foundation.

**Supporting evidence:**
- koboi/harness/env.py:79,82-89 (verified) — `_env_defaults: dict = {}` module-global singleton + configure_env_defaults(), with a code comment noting it is populated once at startup
- koboi/orchestration/ (router/orchestrator/factory) (verified via project CLAUDE.md) — native-only multi-agent, no A2A surface

- Community signal (report, tag as research-sourced and speculative): A2A 24.6k stars; 'OpenAI rejects A2A PR' Oct-2025; 'Ask HN: Is anyone using A2A?' Jun-2026 96 pts (adoption genuinely uncertain)

**Expected customer impact.** Multi-agent orchestration that is correctly isolated in-process and optionally interop-ready with external A2A agents via an agent-card.

**Expected business impact.** A mindshare bet on an unsettled protocol (low cost, high optionality) plus a concrete correctness fix that makes the multi-agent claim defensible.

**Risks:** A2A adoption is genuinely uncertain — the bet may not pay off; mitigate by shipping the isolation fix as the load-bearing deliverable and A2A as opt-in behind a flag; A2A spec churn — track the upstream spec and keep the surface thin (agent-card + task-send/streaming first); Removing the _env_defaults singleton may require plumbing config through every previously-global call site

**Success metrics:** env.py _env_defaults singleton removed; a multi-agent-in-one-process isolation test passes (per-agent env config does not leak); An A2A agent-card interoperates with a reference A2A client (task-send + streaming round-trip); Orchestration mode is documented as production-supported (no longer 'v1 / unsupported resume' caveat on the isolation axis)

> **Adversarial score** — impact **4/10**, effort **6/10**, ROI **0.7**, confidence **0.6**, strategic alignment **4/10** (short-term 3, long-term 4).

> **Biggest risk:** Bundling a narrow ~2PW correctness fix with a speculative protocol bet means the A2A portion either drags the fix's timeline or ships half-baked against a churning spec, while drawing engineering attention away from the wedge and gaps that actually move adoption — the two deliverables should be split and the A2A half deferred until the protocol consolidates.
